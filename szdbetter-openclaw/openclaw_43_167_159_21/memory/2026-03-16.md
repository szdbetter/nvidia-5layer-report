### 根因链
1. `ov.conf` 的 VLM 配置指向 Fiona Ollama (`100.101.75.22:11434`)，但 Fiona 上只有 `nomic-embed-text` embedding 模型，没有任何 LLM → 抽取请求无可用模型 → `extracted 0 memories`

4. 改为 `/api/coding/v3` endpoint 后，抽取成功，返回 1 memory ✅

### 最终 ov.conf VLM 配置
```json
"vlm": {
    "provider": "openai",
    "api_key": "<volcengine_key>",
    "model": "doubao-seed-2.0-lite",
    "api_base": "https://ark.cn-beijing.volces.com/api/coding/v3",
    "max_concurrent": 2
}
```

### 关键教训
- 火山引擎标准 `/api/v3` 不支持 `doubao-seed-2.0-lite`，必须用 `/api/coding/v3`
- OpenViking VLM 错误只记录在 `/tmp/ov.log`（systemd StandardOutput），不在 journalctl 主日志中
- `time: 0.0` + `result: []` 不一定是超时，可能是 LLM 调用失败后静默返回空

### VLM 调用频率与成本分析
- **auto-capture**: 每轮对话结束触发一次（`agent_end` 事件），经过长度/命令/subagent等过滤后调 VLM
- **memory_store**: 手动调用时触发
- **资源入库 (add_resource)**: 每个文件需 L0+L1 生成，大文件分 chunk 多次调用 — 这是烧钱主因
- `code_summary_mode` 默认 `ast`（tree-sitter 解析，不调 LLM），当前已是安全状态
- 用户之前用 Gemini 2.0 Flash 1-2天烧了 600 港币，主因可能是资源批量入库

### 技术选型结论

### Chrome 146 debug 模式启动方式（已验证）
```bash
pkill -9 'Google Chrome'
sleep 2
cp -r ~/Library/Application\ Support/Google/Chrome /tmp/chrome_debug_profile
# 忽略 SingletonSocket/Lock 的 cp 报错
nohup '/Applications/Google Chrome.app/Contents/MacOS/Google Chrome' \
  --remote-debugging-port=9222 \
  --user-data-dir=/tmp/chrome_debug_profile \
  > /tmp/chrome_debug.log 2>&1 &
```
- Chrome 146 强制要求 `--user-data-dir` **不能是默认路径**，否则报错退出
- `open -a` 方式无法传参数，必须用完整路径
- 复制 profile 后登录态（Cookie）会保留，闲鱼可直接使用

### OpenClaw browser 工具接入 Fiona Chrome
- 通过 `node=fiona-mbp2015` 参数路由到 Fiona
- OpenClaw 管理的 Chrome 监听 port 1990（独立于用户 Chrome 的 9222）
- 用户 Chrome 通过 `--remote-debugging-port=9222` 开启后，需确认 OpenClaw gateway 配置指向正确端口
