# OpenClaw 本地 Dashboard 系统设计草案

## 1. 产品定位
Dashboard 不是展示页，而是本地统一任务作战中枢。目标是统一管理 Mission Card、任务状态、Agent/节点执行流、证据、风险阻塞、长期记忆沉淀与验证。

## 2. 核心目标
1. 所有复杂任务先立项再执行。
2. 所有执行过程可追踪、可审计。
3. 所有完成结果必须附证据。
4. 所有核心方法论和结论必须回写 OpenViking。
5. 所有系统入口统一收敛到 Gateway 1979 单入口。

## 3. 用户角色
- VC / 人类合伙人：看目标、看状态、看风险、看 ROI。
- Reese：任务主控、任务拆解、验收、升级。
- Fiona：高频扫描、健康检查、流水线执行。
- ACP / SAP 子智能体：执行垂直任务并提交证据。

## 4. 页面结构
### 4.1 任务总览页
字段：任务名、Mission ID、Owner、优先级、状态、开始时间、最近更新时间、当前阶段、下一动作、风险等级。

### 4.2 Mission Card 详情页
展示任务目标、交付物、Deadline、验收标准、资源清单、执行架构、风险、阻塞、证据、记忆状态。

### 4.3 Agent / 节点执行流页
展示 sessions、subagents、ACP 线程、process、cron、nodes 的运行状态，包括最后动作、最后产物、是否超时、是否异常、关联任务。

### 4.4 证据面板
集中展示文件、日志、截图、数据样本、PR/Issue/Run ID、告警记录、报告链接。

### 4.5 风险与阻塞页
展示卡点、已尝试方案、需要授权点、升级状态、协作文档引用。

### 4.6 Memory 面板
展示 OpenViking 写入状态、最近写入内容摘要、Recall 查询词、Recall 命中结果、SOP 版本。

## 5. 数据来源
- 项目文档：memory/projects/<project>/
- sessions_list / sessions_history
- subagents
- process
- cron
- nodes
- memory_recall
- 本地 artifacts/logs

## 6. 最小功能范围（MVP）
1. 任务列表与过滤
2. Mission Card 详情页
3. 任务状态流转可视化
4. Agent/节点状态聚合
5. 证据面板
6. Memory 写入/Recall 验证面板
7. 风险阻塞面板

## 7. 非目标（MVP 不做）
- 不在第一版内做深度权限系统
- 不在第一版内做复杂多租户
- 不在第一版内深度改造 Gateway 核心
- 不在第一版内开放高风险自动执行按钮

## 8. 架构原则
1. Dashboard 独立服务
2. Gateway 继续做核心控制面
3. 通过反向代理实现 1979 单入口
4. 默认只读
5. 高风险写操作需二次确认
6. 所有关键动作要有审计日志

## 9. 与 Fiona / Polymarket 的结合
Dashboard 应支持将 Fiona 高频扫描任务纳入统一视图：
- 盘口抓取状态
- 规则抽取状态
- 候选池数量
- 高风险模糊性标签
- 告警历史
- 二次复核结果
这样才能把“扫描—复核—沉淀”做成闭环。

## 10. 验收标准
若 Dashboard 方案不能同时回答以下问题，则视为不合格：
- 当前有哪些任务在跑？
- 每个任务的目标和完成标准是什么？
- 谁在执行？
- 卡在哪？
- 证据在哪？
- 是否已写入 OpenViking？
- Recall 是否能找回？
