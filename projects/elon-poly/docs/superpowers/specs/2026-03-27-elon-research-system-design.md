# Elon 推文 Polymarket 研究系统设计

日期：2026-03-27
状态：待用户审阅
范围：仅限研究与验证

## 1. 目标

构建一套全新的 Polymarket「Elon Musk 每周推文数」市场研究系统，用可辩护的证据回答一个核心问题：

是否能够基于历史成交、历史价格时间序列以及带时间戳的推文计数事件，构造出一套人工可执行、具有正期望收益的预测套利策略？

该系统不是交易机器人。它的职责是证明或证伪可交易性。

## 2. 非目标

本阶段不包含以下内容：

- 自动下单执行
- 持仓管理
- 实时止损或熔断
- 监控 daemon
- 可视化 dashboard
- Iran ceasefire、LP farming 或其他 Polymarket 策略支持
- 复用旧执行架构作为新系统基础

## 3. 为什么必须重开

现有 `elon-poly` 项目把研究、执行、监控和有状态交易逻辑混在同一个目录中。当前材料已经证明存在两类核心失败：

1. 策略验证不可信。
   之前高 Sharpe 的回测依赖模拟数据或时间不重叠的数据，无法证明真实历史 edge。

2. 实盘执行架构不安全。
   持仓状态与链上真实状态脱节，止损逻辑不独立，cron 驱动的自动化会引入不一致系统状态。

因此，新系统只把当前目录作为文档和原始数据来源使用。真正的新实现应在下一阶段落到一个干净的新项目中。

## 4. 成功标准

第一阶段只有在同时产出以下结果时才算成功：

- 一套可重复生成的 Elon 周市场历史数据集
- 跨周、跨 bracket 统一标准化后的市场数据
- 与市场时间线对齐的带时间戳推文事件数据
- 能回答是否存在滞后反应或错价的事件研究结果
- 至少两类候选策略的严格回放回测结果
- 一份明确结论，且只能是以下两种之一：
  - 证据支持进入半自动交易设计阶段
  - edge 不足，项目应终止

允许第一阶段得出“不值得交易”的结论。

## 5. 验证标准

本阶段的验证标准弱于“完整历史订单簿重建”，但仍需足够严格，能够支撑真实交易决策。

允许使用的验证输入：

- 历史成交数据
- 历史价格时间序列
- 市场元数据与最终结算结果
- 带时间戳的推文或推文计数事件流

明确不要求：

- 所有历史周的完整订单簿深度

因此，凡是只依赖成交和价格时间序列的历史区间，回测都必须采用保守执行假设。

## 6. 推荐方案

采用“研究系统 + 严格回放回测器”的方案。

曾考虑过三条路线：

1. 仅做事件研究平台
2. 研究系统 + 严格回放回测器
3. 端到端准生产平台

推荐第 2 条。第 1 条不足以支撑交易决策；第 3 条会过早引入旧项目已经证明有害的复杂度。

## 7. 系统边界

新系统只回答：

“在时点 T 当时可见的信息下，市场是否低估了 Elon 推文计数相关信息，或者对其反应失真到足以支持一笔交易？”

新系统不回答：

- 如何自动化执行实盘
- 如何实时管理库存或仓位
- 如何进行实时风险监控
- 如何支持多个互不相关的 Polymarket 子策略

## 8. 架构设计

系统被拆成五个单向流动的层级。

### 8.1 原始采集层

目的：
采集原始来源数据并落盘，不包含任何策略逻辑。

必须覆盖的数据源：

- Elon 每周市场的 Polymarket 市场元数据
- 各 bracket 对应市场或 token 的历史成交
- 各 bracket 对应市场或 token 的历史价格时间序列
- Elon 的逐条推文时间线或推文计数事件时间线
- 最终市场结算结果
- 通过网页搜索、公开数据集、归档站点或第三方导出获取的 Elon 历史推文数据

对历史推文数据的额外要求：

- 每份数据都必须记录来源、抓取时间、覆盖区间、字段说明和已知缺口
- 能下载原始文件时优先下载并保存原始文件，而不是只保留转录结果
- 若某份历史推文数据无法证明其原始时间戳可信，则只能作为辅助参考，不得进入主回测数据集

输出：

- 不可变的原始快照
- 包含来源、采集时间、状态的采集日志
- 原始文件与清洗后文件之间的映射关系

### 8.2 标准化层

目的：
把每周市场转换成统一 schema，确保跨周比较是安全的。

这一层必须解决：

- 不同周的 bracket 区间不同
- 不同 token / asset id 不同
- 时区缺失或不一致
- 多个来源对同一概念的字段命名不同

标准化实体至少应包含：

- 市场周标识
- 事件起止时间
- bracket 下界与上界
- token 或 asset id
- 时间戳
- 价格观测
- 成交观测
- 推文事件或累计计数状态
- 最终结算 bracket

这是强制层。特征工程与回测逻辑不得直接读取原始来源格式。

### 8.3 特征层

目的：
生成研究特征，但不生成交易动作。

第一版最少要支持：

- 时点 T 的累计推文数
- 5 分钟、15 分钟、1 小时、6 小时推文增量
- burst 强度指标
- 距离市场结束剩余时间
- 市场隐含中心与分散度
- 各 bracket 的隐含概率
- 推文事件后的价格响应滞后特征
- 模型公允价值与市场观察价格的偏差

这一层应同时服务于事件研究和策略回放，避免重复实现逻辑。

### 8.4 回放回测层

目的：
只基于每个时间点之前可见的信息，回放候选策略。

约束：

- 禁止未来数据泄漏
- 模拟过程中禁止直接读取结算后的信息
- 执行假设必须显式声明
- 必须处理成交稀疏或中间报价缺失的情况

回测器必须支持三种执行模式：

1. 研究级
   用于快速筛选策略，允许适度简化假设。

2. 保守级
   默认决策模式。加入滑点、成交限制与仓位上限。

3. 杀伤测试级
   采用极保守假设。若 edge 在该模式下消失，则策略不得进入下一阶段。

### 8.5 评估层

目的：
输出可用于决策的证据。

最少输出项：

- 分周收益
- 总收益
- 最大回撤
- 胜率
- 平均持有时长
- 换手率
- 资金使用率
- 对滑点假设的敏感性
- 样本内 / 样本外结果
- 按市场 regime 或推文活跃度 regime 的分层结果

## 9. 策略研究顺序

第一版不从复杂机器学习模型开始，而是优先验证少量可解释策略族。

### 9.1 S1：计数偏差策略

根据带时间戳的累计推文路径估计最终推文总数分布，将其映射为各 bracket 公允概率，再与市场价格比较。

假设：
市场在给定当前推文计数信息时，系统性低估或高估最终 bracket 概率。

### 9.2 S2：事件冲击策略

测量推文 burst 是否会引发 bracket 价格的延迟反应或短期过冲，并测试动量延续或均值回归入场规则。

假设：
市场对推文事件的反应存在可利用的滞后或过度反应。

### 9.3 S3：分布错位策略

比较市场隐含 bracket 分布与历史条件分布，在整条分布发生系统性平移或压缩时进行交易。

假设：
即使单个 bracket 看起来不明显错价，整条分布仍可能出现可交易的结构性偏移。

## 10. 研究流程

每个候选策略都必须按以下顺序通过：

1. 数据源有效性验证
2. 标准化结果验证
3. 事件研究
4. 回放回测
5. 样本外确认
6. 形成推荐结论

任何候选策略都不得跳过事件研究阶段，也不得仅凭样本内回测表现进入下一阶段。

## 11. 回测原则

回测必须正视当前已知数据限制。

规则如下：

- 只使用每个决策时点之前可见的历史成交、价格时间序列和推文时间戳数据
- 每一周视为独立市场，不跨周混算
- 尽量按可观测成交价或价格序列执行，禁止理想化最佳成交价假设
- 当缺少深度数据时，必须加入保守滑点和成交规模上限
- 不得在未完成 canonical mapping 的情况下混合不同 bracket 结构的周数据
- 训练集、验证集、测试集必须按时间切分，不能随机打散
- 参数搜索过程和失败实验都必须记录

任何只在乐观假设下盈利的策略，都视为不可交易。

## 12. 数据源策略

应优先使用官方或直接来源数据，仅在确有历史覆盖缺口时引入第三方数据集。

优先级：

1. Polymarket 官方市场元数据
2. Polymarket 官方历史成交接口或等价公开成交来源
3. 官方价格历史接口
4. 从现在开始自建实时采集归档，补充未来历史
5. 用第三方历史数据集补全过去无法覆盖的窗口

所有第三方导入数据都必须标记来源、覆盖时间范围和可信度说明。

## 13. 防止事后信息泄漏与伪验证规则

历史数据越丰富，越容易在不自觉的情况下把已知结果重新带回特征、规则或参数中。因此该系统必须执行以下硬规则：

1. 点时可得性原则
   每个决策时点只能使用该时点理论上可见的信息。
   可以使用时点 T 之前的推文事件、成交和价格。
   不可以使用该周最终推文总数、最终结算 bracket 或最终结算价格来构造时点 T 的特征。

2. 特征与标签物理分离
   用于建模和生成信号的特征表，与用于结算和评估的标签表必须物理分开存储。
   特征表只允许包含时点 T 及之前的数据。
   标签表只允许在回测结算阶段读取。

3. 冻结数据集版本
   每一轮正式实验都必须基于一个冻结的数据集版本运行。
   数据集版本必须记录生成脚本、覆盖时间窗口和来源列表。
   不允许在看完结果后补数据、改 schema 或重写字段，再声称属于同一轮验证。

4. 预先声明策略规则
   正式回测前必须先写清楚入场条件、出场条件、滑点规则、资金约束和参数范围。
   允许后续迭代，但每次迭代都必须形成新的实验版本记录。

5. 时间顺序切分
   训练集、验证集、测试集必须按时间顺序切分，禁止随机打散。
   任何参数调整都只能基于训练集和验证集完成，最终测试集只允许用于最终确认。

6. 辅助数据不得冒充主证据
   无法确认原始时间戳可信性的网页搜索数据、二次整理数据或人工汇总数据，只能作为探索性参考。
   主回测、主结论和是否晋级的判断，只能基于满足时间戳可信性要求的数据集。

## 14. 目录与项目结构

旧目录继续保留，但只作为参考。

下一阶段创建的新项目建议采用如下结构：

```text
elon-research/
  docs/
  datasets/
  data_sources/
  normalization/
  features/
  strategies/
  backtesting/
  evaluation/
  experiments/
```

关键规则：

- 不包含交易 daemon 代码
- 不包含本地持仓状态机
- 不包含混合用途的临时脚本堆
- 原始采集与策略逻辑不得直接耦合

## 15. 旧项目迁移策略

允许迁移：

- 经 provenance 审查后仍有价值的原始历史数据文件
- 说明性文档
- 已确认有用的市场标识信息
- 已确认有用的来源接口和凭证知识

不得作为架构继承：

- `daemon.py`
- `trader.py`
- `monitor.py`
- `positions.json` 工作流
- 旧 cron 模式
- 旧策略调度逻辑

如果旧项目中的某个算法被证明仍有价值，也必须基于新标准化接口重新实现，而不是原样复制。

## 16. 风险

主要风险包括：

- 历史成交覆盖不完整
- 价格时间序列粒度不足以支持部分事件驱动策略
- 推文时间戳需要跨来源重建或对齐
- 跨周 bracket 标准化比预期更复杂
- 在保守成交假设下 edge 消失

这些风险是可接受的。该项目的意义就是尽早暴露它们，而不是被自动化系统掩盖。

## 17. 阶段完成决策门

本阶段结束时必须给出且只能给出以下两种决定之一：

1. 进入半自动交易系统设计阶段
2. 因证据不足以支持可交易性而终止项目

不允许出现第三种模糊结论。“以后再收集更多数据看看”不是有效结论，除非它对应一个明确、阻断性的关键数据缺口。

## 18. 下一步

下一阶段是为“研究系统”本身编写详细实施计划。

该计划必须覆盖：

- 创建干净的新项目
- 明确历史数据采集任务
- 明确 canonical schema
- 明确事件研究任务
- 明确回放回测任务
- 明确评估与晋级标准

该计划不得包含自动交易相关工作。
