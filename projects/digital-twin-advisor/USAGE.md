# 数字分身顾问调用协议

## 目标
让多个数字分身顾问可以被快速切换、快速比较、快速联合作答。

## 最短可用口令

默认规则：**直接写人名 = 调用对应顾问**，不需要额外写“分身”两个字。

### 1. 切换到单个顾问
```md
使用【刘小排】回答：<你的问题>
```

### 2. 指定回答模式
```md
使用【刘小排分身】回答这个问题，模式为：
- 先给结论
- 再给理由
- 最后给下一步动作

问题：<你的问题>
```

### 3. 要求严格按顾问风格输出
```md
切换到【刘小排】。
严格按照他的思维模型和表达风格回答，不要用你自己的默认口吻。
问题：<你的问题>
```

## 多顾问快速调用

### 4. 并排比较
```md
让【刘小排】、【顾问B】分别给判断，最后再给我一个差异总结。
问题：<你的问题>
```

### 5. 指定角色分工
```md
围绕这个问题组织一次顾问会：
- 【刘小排分身】负责 AI 产品化和工作流判断
- 【顾问B分身】负责增长/流量判断
- 【顾问C分身】负责商业化判断

问题：<你的问题>
输出：先分别回答，再给联合结论。
```

### 6. 快速筛选最该问谁
```md
针对这个问题，先判断最适合找哪个数字分身顾问回答，并说明原因；
如果需要，也可以让 2-3 个顾问联合作答。

问题：<你的问题>
```

## 推荐提问结构
尽量带上以下四个要素：
1. 我现在要解决什么问题
2. 目标是什么
3. 约束是什么（时间/预算/团队/资源）
4. 希望输出什么格式（结论/方案/步骤/对比）

范式：
```md
使用【某顾问分身】回答。

背景：
目标：
约束：
问题：
输出格式：
```

## 未来规模化建议
当顾问越来越多时，建议固定三层机制：

### A. 顾问注册表
维护一个 registry，至少记录：
- 顾问名
- 一句话定位
- 最擅长问题
- 不适合问题
- 知识截止日期
- 对应 prompt 文件路径

### B. 路由规则
先判断问题属于：
- 产品
- 增长
- 商业化
- 投资
- 效率/组织
- 技术实现

再匹配最适合的 1-3 个顾问。

### C. 默认输出协议
多顾问输出时统一采用：
1. 各顾问独立观点
2. 一致点
3. 分歧点
4. 最终综合建议
5. 下一步动作
